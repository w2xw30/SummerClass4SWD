# Task 5 — Rendering the UI and event handling with JavaScript

## Overview

Right now, `index.html` has the task list and summary counts **hard-coded** directly into the HTML. Your job in this task is to rip that hard-coded markup out and have JavaScript generate it dynamically from data, the same way a real app would.

You will **not** be saving or editing any data yet — that comes in Task 6. For now, you are purely focused on **rendering what already exists in `taskList`**.

By the end of this task, the page should look and behave exactly the same as it does now, except every task card and every summary number is generated by your code, not typed into the HTML.

## What's already there

Open `script.js`. You'll see a global array already defined for you:

```js
const taskList = [
  {
    title: "To do home-work",
    deadline: "09/01/2026",
    description: "This is a new task",
    status: 0,
  },
];
```

Add a few more sample tasks to this array so you have something to test with — make sure you have at least one task in each status. We'll treat `status` as a number:

| Value | Meaning     |
| ----- | ----------- |
| `0`   | In Progress |
| `1`   | Completed   |
| `2`   | Deleted     |

You'll also want to look at `index.html` to see the markup that's currently hard-coded for:

- The 3 summary cards (`#summary-container`)
- The 3 example task cards (`.task-list`)
- The "Add Task" modal (`#add-task-overlay`)

And in `style.css`, take note of these classes — you'll need to apply them yourself once you generate the cards in JS:

- `.task-card`, plus one of `.status-inprogress`, `.status-completed`, `.status-deleted`
- `.status-badge`, plus one of `.badge-inprogress`, `.badge-completed`, `.badge-deleted`
- `.btn`, `.btn-primary`, `.btn-secondary`

## Your tasks

### 1. Remove the hard-coded markup

Delete the 3 hard-coded `<li>` task cards from inside `<ul class="task-list">` in `index.html`, and the 3 hard-coded `<div class="summary-card">` blocks inside `#summary-container`. Leave the empty containers in place — your JS will fill them.

### 2. Render the summary counts

Write a function called `renderSummary` that:

- Counts how many tasks in `taskList` have status `0`, `1`, and `2`. A simple way to do this is to loop over `taskList` and increment a counter for each status value you encounter.
- Inserts those three numbers into the three `.summary-card` elements inside `#summary-container`. Each summary card has a unique `id` attribute (e.g. `inprogress-count`, `completed-count`, `deleted-count`) — use `document.getElementById` to find them and set their `textContent`.

Call this function inside the `DOMContentLoaded` listener so it runs as soon as the page loads.

### 3. Render the task list

Write a function called `renderTaskList` that:

- Selects the `<ul class="task-list">` element and clears its contents by setting `innerHTML = ""`. This ensures you start fresh every time the function runs.
- Loops over `taskList`. For **each task**, builds a `.task-card` element matching the structure of the cards that used to be hard-coded: title, deadline date, status badge, description, and a footer with action buttons. Using an HTML template string (a backtick string with `${}` placeholders) is the simplest approach.
- Applies the correct status class (`status-inprogress`, `status-completed`, or `status-deleted`) and the correct badge class and badge text based on each task's `status` value.
- Creates a new `<li>` element, sets its `innerHTML` to your card HTML, and appends it to the `<ul>`.

Call this function inside the `DOMContentLoaded` listener alongside `renderSummary`.

**A helper function will save you a lot of repetition.** Both this function and later steps need to know what CSS class, badge class, and badge text go with a given status number. Write a small `getStatusInfo(status)` helper that takes a status number and returns an object with those three values — for example:

```js
function getStatusInfo(status) {
  switch (status) {
    case 0:
      return {
        statusClass: "status-inprogress",
        badgeClass: "badge-inprogress",
        badgeText: "In Progress",
      };
    case 1:
      return {
        statusClass: "status-completed",
        badgeClass: "badge-completed",
        badgeText: "Completed",
      };
    case 2:
      return {
        statusClass: "status-deleted",
        badgeClass: "badge-deleted",
        badgeText: "Deleted",
      };
  }
}
```

Then call it from your card-building code instead of writing the same `if`/`switch` in multiple places.

**Conditional buttons — this is the important part.** The footer buttons are _not_ the same for every status:

- An **in-progress** task shows both a "Mark As Complete" button and a "Delete" button.
- A **completed** task shows only a "Delete" button.
- A **deleted** task shows no buttons at all.

Isolate this logic in its own helper, `buildTaskButtonsHTML(id, status)`, that returns the correct button HTML string for a given task. Call it from inside your card template. Don't worry about making the buttons _do_ anything yet — that's the next step.

### 4. Add Mark As Complete and Delete Functionality

Now that the buttons are rendering, wire them up so they actually change task data.

Each button needs a `data-id` attribute set to the task's `id` so the event handler knows which task was clicked. If you followed step 3, your `buildTaskButtonsHTML` helper already receives the `id` — make sure you're injecting it into the button markup like this:

```html
<button data-id="${id}" ...>Mark As Complete</button>
```

Write two event handler functions:

**`markAsCompleteEventHandler(e)`** — called when "Mark As Complete" is clicked:

1. Call `e.stopPropagation()` to prevent the click from bubbling up to other listeners.
2. Read the task id from `e.target.dataset.id`. Note that `dataset` values are always strings, so parse it with `parseInt` before comparing to the numeric ids in `taskList`.
3. Find the matching task in `taskList` using `findIndex`.
4. Set that task's `status` to `1`.
5. Call `renderSummary()` and `renderTaskList()` to update the page.

**`deleteEventHandler(e)`** — called when "Delete" is clicked. Follows the same pattern, but sets the task's `status` to `2`.

The simplest way to attach these handlers is with inline `onclick` attributes directly in the button HTML you build in step 3:

```html
<button ... onclick="markAsCompleteEventHandler(event);">
  Mark As Complete
</button>
```

Because these functions are declared at the top level of your script, they are available on `window` and accessible from inline handlers. Test that clicking "Mark As Complete" on an in-progress task moves it to completed and updates both the card and the summary counts.

### 5. Add functionality to the sidebar buttons (implement filters)

The sidebar contains buttons that let the user filter the task list by status. Each button has a `data-filter` attribute set to the status number it represents (e.g., `data-filter="0"` for In Progress, `data-filter="1"` for Completed), and one button has `data-filter="-1"` to show all tasks.

Write a function called `setupSidebarButtons` that:

1. Selects all buttons inside `.sidebar-menu` using `querySelectorAll`.
2. Attaches a `click` event listener to each one.
3. Inside the handler, reads the filter value from `btn.dataset.filter` and calls `renderTaskList` with it, passing the filter as a number using `parseInt`.
4. Updates the visual "selected" state: find the element that currently has the `selected` class, remove it, then add `selected` to the clicked button's parent `<li>` element.

To support filtering, update `renderTaskList` to accept an optional `filterValue` parameter (defaulting to `-1`). Before building each card, check whether `filterValue === -1` (show all) or the task's `status` matches `filterValue`. Skip tasks that don't match.

Call `setupSidebarButtons()` in the `DOMContentLoaded` listener.

### 6. Add modal open and close event functionality

The "Add Task" button should open a modal form, and clicking the close or cancel button should close it.

The relevant elements in your HTML are:

- `#add-task-btn` — the button that opens the modal
- `#add-task-overlay` — the dark background overlay
- `#add-task-modal` — the modal panel itself
- `#close-modal-btn` — the × button inside the modal
- `#cancel-add-task-btn` — the "Cancel" button inside the modal form

Write a `toggleModal` function that toggles the `hidden` class on both `#add-task-modal` and `#add-task-overlay`. When either element has the `hidden` class applied via CSS it should be invisible; removing it makes it appear. Toggling both at once keeps the overlay and modal in sync.

Write a `setupModalEvents` function that:

1. Attaches a `click` listener to `#add-task-btn` that calls `toggleModal` (use `e.stopPropagation()` here to prevent the window-level click handler from immediately closing the modal).
2. Attaches `click` listeners to both `#close-modal-btn` and `#cancel-add-task-btn` that also call `toggleModal`.

Call `setupModalEvents()` in the `DOMContentLoaded` listener.

### 7. Add form submission from the modal form

Write a function called `setupFormSubmission` that:

1. Selects the `#add-task-form` element.
2. Attaches a `submit` event listener to it.
3. Inside the handler, calls `e.preventDefault()` to stop the browser from reloading the page.
4. Reads the form values using `new FormData(e.target)` and converts them to a plain object with `Object.fromEntries(formData.entries())`. This gives you an object with keys matching each input's `name` attribute (`title`, `deadline`, `description`).
5. Prepends a new task object to `taskList`. The new task needs a unique `id` — a simple approach is to take the current maximum id in the list and add 1: `1 + Math.max(...taskList.map(x => x.id))`. Set `status` to `0` (In Progress) and spread the form data in.
6. Resets the form with `addTaskForm.reset()` so the fields are blank for next time.
7. Calls `renderTaskList()` and `renderSummary()` to update the page.
8. Calls `toggleModal()` to close the modal.

Call `setupFormSubmission()` in the `DOMContentLoaded` listener.

## Things to think about

- Where should your render functions live, and when should they run? (Hint: you already have a `DOMContentLoaded` listener in `script.js`.)
- How will you avoid duplicating your "which classes/badge/buttons go with this status" logic in multiple places? Could one small helper function answer "what does this status look like" so both the card renderer and other features can reuse it?
- Why does `e.stopPropagation()` matter on the "Add Task" button click? Think about what the window-level click handler does and what would happen without it.
- Don't worry about making this efficient: `innerHTML`, `createElement`, or template strings are all fine for this task.
